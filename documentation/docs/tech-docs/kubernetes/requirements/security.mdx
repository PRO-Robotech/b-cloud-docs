# 2.10 Информационная безопасность

## 2.10.1 Требования к идентификации и аутентификации

<table>
  <thead>
    <tr>
      <th>№</th>
      <th>Требование</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>1</td>
      <td>Обеспечивается создание уникальной учетной записи для каждого пользователя</td>
    </tr>
    <tr>
      <td>2</td>
      <td>Механизмы аутентификации реализуются с использованием защищенных протоколов аутентификации. При хранении и передаче конфиденциальность паролей обеспечивается шифрованием или хешированием с применением стойких криптографических алгоритмов</td>
    </tr>
    <tr>
      <td>3</td>
      <td>Обеспечивается возможность аутентификации в Active Directory</td>
    </tr>
    <tr>
      <td>4</td>
      <td>Реализуются механизмы двухфакторной аутентификации пользователей</td>
    </tr>
    <tr>
      <td>5</td>
      <td>Проверка учетных данных для всех учетных записей проводится на стороне серверных компонентов ИС (отсутствие возможности оффлайн аутентификации)</td>
    </tr>
    <tr>
      <td>6</td>
      <td>В процессе аутентификации в ИС пароль не отображается при его вводе, проверка введенной информации (логин, пароль) осуществляется только после полного ее ввода, в случае обнаружения ошибки, система не уточняет, какие именно данные введены неправильно</td>
    </tr>
  </tbody>
</table>

## 2.10.2 Сбор и анализ событий информационной безопасности

<table>
  <thead>
    <tr>
      <th>№</th>
      <th>Требование</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>1</td>
      <td>Обеспечивается возможность регистрации событий безопасности, включая: попытки идентификации, изменения полномочий, модификацию профиля пользователей, действия привилегированных пользователей, доступ к защищаемым объектам</td>
    </tr>
    <tr>
      <td>2</td>
      <td>Обеспечивается фиксация информации для каждого события безопасности: тип события, дата и время, идентификационная информация, результат события, субъект и объект доступа, описание действий</td>
    </tr>
    <tr>
      <td>3</td>
      <td>Обеспечивается ограничение доступа к журналам регистрации событий безопасности только уполномоченным пользователям</td>
    </tr>
  </tbody>
</table>

## 2.10.3 Защита информации при использовании мобильных устройств

<table>
  <thead>
    <tr>
      <th>№</th>
      <th>Требование</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>1</td>
      <td>Ограничение доступа пользователей с использованием разрешенных версий мобильного ПО</td>
    </tr>
    <tr>
      <td>2</td>
      <td>Применение технологии SSL/TLS Pinning для защиты корневых сертификатов</td>
    </tr>
    <tr>
      <td>3</td>
      <td>Использование защищенных хранилищ для хранения аутентификационных данных пользователя</td>
    </tr>
    <tr>
      <td>4</td>
      <td>Журналирование событий в незащищенные области не допускается</td>
    </tr>
    <tr>
      <td>5</td>
      <td>Ограничение передачи конфиденциальной информации между приложениями</td>
    </tr>
  </tbody>
</table>

## 2.10.4 Требования к парольной защите и учетным записям

<table>
  <thead>
    <tr>
      <th>№</th>
      <th>Требование</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>1</td>
      <td>Пользователям обеспечивается возможность самостоятельной установки и смены пароля</td>
    </tr>
    <tr>
      <td>2</td>
      <td>Обеспечивается возможность назначения первичного пароля администратором ИС и обязательной смены пароля пользователем при первичной аутентификации в ИС</td>
    </tr>
    <tr>
      <td>3</td>
      <td>Неиспользуемая в течение 30 (тридцать) дней учетная запись пользователя блокируется в ИС</td>
    </tr>
    <tr>
      <td>4</td>
      <td>Обеспечивается возможность заблаговременного (не менее чем за 14 дней) оповещения пользователей о необходимости смены пароля (посредством сообщений, подсказок или почтовых рассылок)</td>
    </tr>
    <tr>
      <td>5</td>
      <td>Реализована возможность установки срока длительности простоя (не более 15 минут) пользовательской сессии, после которого сессия принудительно блокируется до повторной аутентификации</td>
    </tr>
    <tr>
      <td>6</td>
      <td>Обеспечивается возможность хранения истории паролей (хэшей паролей) пользователей для предотвращения повторного их использования. Проверяется повтор последних 7 паролей</td>
    </tr>
    <tr>
      <td>7</td>
      <td>Обеспечивается возможность установки ограничений на длину и сложность пароля (буквы в верхнем и нижнем регистрах, цифры и специальные символы)</td>
    </tr>
    <tr>
      <td>8</td>
      <td>Обеспечивается возможность блокировки учетной записи на заранее определенный срок после заданного количества неудачных попыток аутентификации</td>
    </tr>
    <tr>
      <td>9</td>
      <td>Обеспечивается принудительная смена пароля через установленный промежуток времени</td>
    </tr>
    <tr>
      <td>10</td>
      <td>Реализованы механизмы усложняющих техник ввода пароля: CAPTCHA, увеличение времени ожидания</td>
    </tr>
  </tbody>
</table>

## 2.10.5 Требования к управлению правами доступа/разделению полномочий

<table>
  <thead>
    <tr>
      <th>№</th>
      <th>Требование</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>1</td>
      <td>Обеспечивается реализация механизмов авторизации пользователей с разделением прав доступа к информации/данным и функциям внутри ИС</td>
    </tr>
    <tr>
      <td>2</td>
      <td>Обеспечивается предоставление пользователям прав доступа на основании атрибутной или ролевой модели</td>
    </tr>
    <tr>
      <td>3</td>
      <td>Обеспечивается явное ограничение или запрет на действия пользователя в ИС до прохождения процедур идентификации и аутентификации</td>
    </tr>
    <tr>
      <td>4</td>
      <td>Запрещён по умолчанию прямой доступ пользователей, администраторов, третьих лиц в продуктивные базы данных ИС</td>
    </tr>
    <tr>
      <td>5</td>
      <td>Хранение и обработка информации Банка во внешних облачных сервисах запрещена</td>
    </tr>
    <tr>
      <td>6</td>
      <td>Осуществляется информирование пользователей ИС перед непосредственным получением доступа к ИС / ИА в формате системного уведомления / баннера. Уведомление или баннер сохраняются на экране, пока пользователи не подтвердят условия использования и примут явные меры для входа в ИС</td>
    </tr>
  </tbody>
</table>

## 2.10.6 Общие требования к информационной безопасности

<table>
  <thead>
    <tr>
      <th>№</th>
      <th>Требование</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>1</td>
      <td>Просмотр сводной информации по политикам безопасности в кластере, в первую очередь Network Policy</td>
    </tr>
    <tr>
      <td>2</td>
      <td>Поддерживает разграничение доступа на основании ролевой (RBAC) или атрибутивной (ABAC) модели</td>
    </tr>
    <tr>
      <td>3</td>
      <td>Поддерживает разграничение доступа на уровне записей/сущностей одного типа (row-level security)</td>
    </tr>
    <tr>
      <td>4</td>
      <td>Имеется механизм шифрования хранимых данных (указать, каких именно данных и требования к алгоритмам шифрования)</td>
    </tr>
    <tr>
      <td>5</td>
      <td>Аудит событий по изменению конфигураций, настроек информационной системы, установки обновлений</td>
    </tr>
    <tr>
      <td>6</td>
      <td>Аудит событий по изменению данных в результате выполнения бизнес-процессов и хронологическую последовательность действий пользователей</td>
    </tr>
    <tr>
      <td>7</td>
      <td>Возможность исключать из записи в журнал событий чувствительные данные, такие как PAN карт и т.п.</td>
    </tr>
  </tbody>
</table>
